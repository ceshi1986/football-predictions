# 足球预测网 - 常犯错误清单

> 每次踩坑后补充，每日自检脚本自动检查其中可自动化的项目。
> 最后更新：2026-07-18

---

## 一、高频必查项

### 1. 前后端修改不同步
**症状**：修改了Python脚本的联赛白名单/逻辑，但忘了同步修改前端index.html中的对应配置。
**案例**：ACTIVE_LEAGUE_CODES只在Python端加了，前端`_activeLeagues`还是旧的，命中率前后端不一致。
**防范**：涉及联赛列表/白名单的修改，必须同时改：
- `daily_predictions.py` → ACTIVE_LEAGUE_CODES
- `index.html` → `_activeLeagues`（至少2处：赛程加载 + 命中率面板）
- `fetch_schedule.py` → LEAGUES列表
- `fetch_odds_api.py` → SPORT_KEYS映射

### 2. 命中率统计范围不一致
**症状**：脚本端和前端计算的命中率数值不同。
**案例**：脚本端按ACTIVE_LEAGUE_CODES过滤了验证数据，但前端_updateAccPanel没同步过滤。
**防范**：命中率过滤逻辑必须前后端同步修改。

### 3. 联赛清理不彻底
**症状**：去掉某联赛时只改了部分文件，其他文件还残留配置。
**防范**：去掉联赛时grep全项目确认无残留：`grep -r "联赛名" .`

## 二、数据与预测准确性

### 4. 中文显示问题
**症状**：联赛名、球队名显示为英文代码或乱码，前端用户看不懂。
**案例**：schedule.json中联赛名用了wcup2026.org的英文原始名，前端没有正确映射到中文。
**防范**：
- 新增联赛时必须同步添加中文名映射（index.html中的leagueNames）
- 球队中文名需要在schedule_name_map中维护
- 每次新增联赛后检查前端是否能正确显示中文

### 5. 未拉取实时真实数据来预测
**症状**：预测时用的是旧的赔率数据或根本没有赔率数据，导致预测质量下降。
**案例**：fetch_odds_api.py没有正常运行，daily_predictions.py回退到纯Elo降级预测。
**防范**：
- 自检脚本检查odds_api_odds.json的最后更新时间
- 如果赔率数据超过24小时未更新，告警
- daily_predictions.py执行时应验证赔率数据时效性

### 6. 赔率错误/映射错误
**症状**：赔率数据与比赛不匹配，或Odds API的sport_key映射错误导致拉错数据。
**案例**：
- 新增联赛时SPORT_KEYS映射写错，拉到错误联赛的赔率
- 多个bookmaker的赔率取均值时逻辑有bug
- 平局赔率估算逻辑有误
**防范**：
- 新增sport_key映射后验证该联赛赔率是否合理（胜平赔率范围检查）
- 自检脚本抽样检查赔率数据合理性（赔率应在1.01-50之间，三项之和应>1）

### 7. 主客队弄反
**症状**：预测结果的主客队与实际赛程相反，导致用户投注方向完全错误。
**案例**：
- wcup2026.org赛程中home/away字段与Odds API不一致
- 赔率匹配时按球队名匹配，但Odds API的home/away顺序与赛程不同
- 球队名模糊匹配时匹配到了错误的球队，导致赔率张冠李戴
**防范**：
- 赔率匹配后必须校验：匹配到的赔率中的home球队=赛程中的home球队
- 如果赔率中的主客场与赛程相反，需要翻转赔率（胜→负互换）
- 自检脚本抽查：验证schedule.json中前几场比赛的赔率主客队是否与赛程一致

### 8. 手动预测和AI预测不一致
**症状**：用户手动点击预测的结果与定时脚本自动生成的结果不同。
**案例**：
- 手动预测使用前端JS计算，定时脚本使用Python计算，两者算法实现有差异
- 赔率数据来源不同：手动预测读取时赔率已更新，脚本执行时用的是旧赔率
- 随机种子不同导致相同输入产生不同输出
**防范**：
- 前端JS的_calc函数与Python的预测逻辑必须保持算法一致
- 手动预测应读取同一份ai-predictions.json，而不是重新计算
- 如果前端有独立计算逻辑，修改算法时前后端同步更新

### 9. 球队名称匹配失败
**症状**：命中率验证时球队名匹配不上，导致已完赛的比赛无法验证。
**案例**：wcup2026.org的球队名带重音符号（如"Déportivo"），Odds API是纯英文，两边对不上。
**防范**：球队匹配逻辑使用Unicode重音归一化 + 英文模糊匹配 + schedule_name_map人工维护。

## 三、发布与部署

### 10. 修改后未推送到GitHub
**症状**：代码改了但忘了git commit + push，线上还是旧版。
**防范**：修改完成后立即推送，自检脚本检查线上版本是否反映最新修改。

### 11. GitHub Pages缓存
**症状**：代码已推送但网页没更新，GitHub Pages有CDN缓存延迟。
**防范**：推送后等待几分钟再验证，或在URL后加时间戳强制刷新。
